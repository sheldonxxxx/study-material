### Zep Paging Utility (zep_paging.py)

**Problem Solved:** Zep's list APIs use UUID cursor pagination. This utility provides transparent complete list retrieval.

```python
# Line 20-23: Constants
_DEFAULT_PAGE_SIZE = 100
_MAX_NODES = 2000
_DEFAULT_MAX_RETRIES = 3
_DEFAULT_RETRY_DELAY = 2.0  # Exponential backoff

# Line 59-102: fetch_all_nodes
def fetch_all_nodes(client: Zep, graph_id: str, page_size: int = 100, max_items: int = 2000):
    all_nodes = []
    cursor = None
    while True:
        kwargs = {"limit": page_size}
        if cursor:
            kwargs["uuid_cursor"] = cursor
        batch = _fetch_page_with_retry(client.graph.node.get_by_graph_id, graph_id, **kwargs)
        if not batch:
            break
        all_nodes.extend(batch)
        if len(all_nodes) >= max_items:
            all_nodes = all_nodes[:max_items]
            break
        cursor = getattr(batch[-1], "uuid_", None) or getattr(batch[-1], "uuid", None)
```

def _call_with_retry(self, func, operation_name, max_retries=3, initial_delay=2.0):
    last_exception = None
    delay = initial_delay
    for attempt in range(max_retries):
        try:
            return func()
        except Exception as e:
            last_exception = e
            if attempt < max_retries - 1:
                time.sleep(delay)
                delay *= 2  # Exponential backoff
            else:
                logger.error(...)

## Cross-Cutting Observations

### Technical Debt / Issues

1. **Zep API Attribute Naming Inconsistency**
   - Files use both `uuid_` and `uuid` interchangeably
   - Workaround via `getattr(node, 'uuid_', None) or getattr(node, 'uuid', None)`
   - Indicates Zep SDK or API version inconsistency

2. **Simulation Environment Dependency**
   - `interview_agents` requires OASIS to be running
   - Failure modes are handled but rely on catching `ValueError`
   - No health check endpoint for OASIS availability

3. **Chat History Memory**
   - `Step5Interaction` caches chat per agent in `chatHistoryCache`
   - No persistence across page refreshes
   - Large chat histories could cause memory issues

4. **GraphPanel Performance**
   - Force simulation on large graphs (2000+ nodes) may lag
   - No virtualization or clustering for large datasets
   - `nodeMap` rebuilt on every filter operation

### Security Considerations

1. **UUID Exposure**
   - Node/edge UUIDs exposed in UI detail panels
   - No access control on graph data visibility

2. **Interview Agent Prompt Injection**
   - `_clean_tool_call_response` does regex-based JSON extraction
   - Could potentially be bypassed with malformed JSON

### Integration Patterns

1. **Polling-Based Real-Time Updates**
   - 2-3 second polling intervals throughout
   - No WebSocket or SSE for true real-time
   - Works but inefficient for high-frequency updates

2. **Retry with Exponential Backoff**
   - Consistent pattern in both frontend (API calls) and backend (Zep)
   - Standard industry practice

3. **Fallback/Degradation**
   - Zep search falls back to local keyword search
   - Interview falls back to error message if OASIS unavailable
   - Good resilience pattern

### Notable Design Decisions

1. **Dual Platform (Twitter + Reddit)**
   - Consistent separation throughout workflow
   - Platform-specific actions and UI styling
   - Dual-API calls in parallel

2. **5-Step Wizard with State Persistence**
   - Each step is a separate component
   - State flows via props and router params
   - Clear phase transitions

3. **Report Agent Tools as First-Class UI**
   - Tools displayed with icons, descriptions
   - Interview results rendered as structured cards
   - Good UX for understanding agent reasoning
