# agency-agents Repository Architecture

## Overview

**Repository Type:** AI Agent Definition Knowledge Base
**Architecture Pattern:** Content-Driven Multi-Format Publishing System
**Core Function:** Collection of specialized AI agent personalities (markdown definitions) with automated conversion to multiple AI coding tool formats

This is not a traditional software codebase. The "architecture" is a **content publishing system** where the product is agent definitions (markdown files) and the build output is tool-specific integration formats.

## Key Abstractions

### 1. Agent Definition Schema

Every agent follows a standardized markdown structure with YAML frontmatter:

```yaml
---
name: Agent Name
description: One-line description
color: colorname or "#hexcode"
emoji: 🎯
vibe: One-line personality hook
services:                    # optional
  - name: Service Name
    url: https://...
    tier: free|freemium|paid
---
```

**Body sections** (grouped by semantic purpose):

| Persona (Identity) | Operations (Behavior) |
|-------------------|----------------------|
| Identity & Memory | Core Mission |
| Communication Style | Technical Deliverables |
| Critical Rules | Workflow Process |
| | Success Metrics |
| | Advanced Capabilities |

### 2. Domain Taxonomy

Agents are organized by **business domain**, not technical function:

| Domain | Count | Focus |
|--------|-------|-------|
| marketing | 29 | Growth, content, social media |
| engineering | 25 | Software development |
| specialized | 27 | Niche roles (MCP, QA, etc.) |
| game-development | 12+ | Subdivided by engine |
| design | 8 | UI/UX, visual design |
| Others | ~90 | academic, product, sales, etc. |

**Rationale:** Domain organization makes it intuitive for humans to discover agents. Technical function would be harder to navigate for non-technical stakeholders.

### 3. Tool Integration Layer

The `integrations/` directory contains output generated by `convert.sh`. Each tool has different requirements:

| Tool | Output Format | Conversion Strategy |
|------|--------------|---------------------|
| Claude Code | Native `.md` + frontmatter | None (direct use) |
| Cursor | `.mdc` rules | Strip frontmatter to description + globs |
| Aider | Single `CONVENTIONS.md` | Accumulate all agents |
| Windsurf | Single `.windsurfrules` | Accumulate all agents |
| OpenClaw | `SOUL.md` + `AGENTS.md` + `IDENTITY.md` | Section splitting by header keywords |
| OpenCode | `.md` agents | Color normalization to hex |
| Gemini CLI | `skills/*.md` | Minimal frontmatter |
| Antigravity | `SKILL.md` per agent | Add risk/source metadata |
| Qwen | SubAgent `.md` files | `${variable}` templating support |

## Component Analysis

### convert.sh — The Core Pipeline

**Responsibility:** Transform canonical agent markdown into tool-specific formats.

**Key design patterns:**

1. **Frontmatter extraction helpers:**
   - `get_field <field> <file>` — Extract YAML field
   - `get_body <file>` — Strip frontmatter, return body
   - `slugify <name>` — Human name to kebab-case slug

2. **Color normalization:**
   - Named colors (cyan, blue, etc.) mapped to `#RRGGBB`
   - Validation and fallback to `#6B7280` (gray)

3. **Section splitting (OpenClaw):**
   - Headers with keywords (`identity`, `communication`, `rules`) -> SOUL.md
   - All other headers -> AGENTS.md
   - IDENTITY.md derived from frontmatter (emoji, name, vibe)

4. **Accumulation pattern (Aider/Windsurf):**
   - Single-file outputs accumulate into temp files
   - Written at end of all conversions

5. **Parallel execution:**
   - Independent tools run via `xargs -P`
   - Buffered output to maintain per-tool grouping

### Agent Definition Files

**Structure pattern:**
- YAML frontmatter (machine-readable metadata)
- Markdown body (human-readable content)
- Section headers define semantics (## Identity, ## Mission, etc.)

**Design principles applied:**
1. Strong personality (distinct voice, not generic)
2. Clear deliverables (concrete code/templates)
3. Success metrics (measurable outcomes)
4. Proven workflows (step-by-step processes)
5. Learning memory (what agent remembers)

### Workflow Examples

Demonstrates multi-agent coordination patterns:

| Pattern | Example |
|---------|---------|
| Sequential handoffs | Sprint Prioritizer -> Backend Architect |
| Parallel activation | UX Researcher + Sprint Prioritizer (Week 1) |
| Quality gates | Reality Checker at midpoint and launch |
| Context passing | Full output pasted between agents |

## Architectural Observations

### Strengths

1. **Canonical single source:** One agent definition, many formats
2. **Human-first organization:** Domain-based discovery for contributors
3. **Tool-agnostic core:** Agent content doesn't assume any specific tool
4. **CI validation:** Linting ensures consistency across 193+ files
5. **Clear separation:** Persona vs Operations allows tool-specific splitting

### Constraints

1. **No runtime state:** Agents are static definitions, no execution model
2. **No agent-to-agent communication:** Workflows are documented patterns, not enforced
3. **Manual installation:** `install.sh` copies files but doesn't manage updates
4. **Git-ignored output:** Integrations directory not versioned — users run conversion locally

### Extension Points

1. **New tool support:** Add `convert_<tool>()` function + case in main loop
2. **New domain:** Create directory, add to `AGENT_DIRS` array
3. **New agent property:** Add to frontmatter schema, update converters
4. **Workflow automation:** Examples could be encoded in a machine-readable format
